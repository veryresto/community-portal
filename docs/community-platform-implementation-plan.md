# Community Platform Implementation Plan

This document defines the strategy for safely and incrementally extracting the identity, approval, and permission system from the IPL Finder codebase into a dedicated, reusable shared community identity platform (`community-platform`).

## 1. Target Architecture

The future architecture relies on a centralized identity provider sharing a Supabase backend with multiple independent applications.

### Application Ownership Model
To ensure future residents can safely build their own apps and teams can evolve independently, there is a strict separation of concerns:

**Applications are responsible for:**
- defining their own capabilities
- maintaining their own business logic
- securing their own tables via RLS

**The platform (`community-platform`) is responsible for:**
- assigning capabilities to users
- resident approval
- global governance lifecycle

### Shared Platform: `community-platform` (`community.veryresto.com`)
This application acts as the single source of truth for identity and governance.
**Responsibilities:**
- **Authentication Entrypoint**: Centralized login via Google OAuth.
- **Waiting Room**: Collects mandatory profile information (e.g., house number) for new users.
- **Resident Approval**: The global dashboard where admins approve/reject new residents.
- **Permission Assignment**: The governance center to assign app-specific capabilities (e.g., granting a user `ipl_finder.upload` or `kas.manage`).
- **Shared Profiles**: The master source for user avatars, names, and contact methods.

### Independent Applications (e.g., IPL Finder, Rekap, Surat, Kas)
Each consumer application remains fundamentally autonomous.
**Responsibilities:**
- **Independently Deployable**: Own separate codebases, repositories, and CI/CD pipelines.
- **Shared Session**: Seamlessly consume the Supabase session generated by `community-platform` via subdomain cookie sharing (`*.veryresto.com`).
- **Centralized Identity**: Read from the shared `profiles` and `user_app_permissions` tables for UI state and RLS enforcement.
- **App-Specific Logic**: Maintain domain-specific tables (e.g., `files`, `transactions`) and UI components without worrying about the heavy lifting of user governance.

## 2. Extraction Strategy

To prioritize minimal disruption and operational safety, the extraction will follow a phased approach, temporarily tolerating data duplication to avoid a "big bang" migration.

### Phase 1: Scaffold `community-platform`
- **Objective**: Create the new repository and deploy the baseline framework.
- **Scope**: Next.js or Vite React scaffold, Tailwind CSS, Supabase client connection pointing to the *existing* IPL Finder Supabase project.
- **Migration Risk**: None. Sits safely alongside the production IPL Finder.
- **Rollback**: Delete the deployed scaffold.

### Phase 2: Move Waiting-Room UI
- **Objective**: Implement the `PendingApprovalScreen` and `RejectedScreen` in `community-platform`.
- **Scope**: Recreate the UI for gathering `house_number` and displaying rejection status.
- **Migration Risk**: Low. These screens run parallel to the ones in IPL Finder.

### Phase 3: Move Approval/Admin UI
- **Objective**: Create the global Admin Dashboard in `community-platform`.
- **Scope**: Replicate the `Admin.tsx` functionality to list users, approve them, and manage the existing `user_roles` and `user_permissions`.
- **Migration Risk**: Low. Both dashboards can co-exist and modify the same underlying tables temporarily.

### Phase 4: Centralize Permissions
- **Objective**: Introduce the new scalable permission architecture (see Section 4).
- **Scope**: Create `applications`, `app_permissions`, and `user_app_permissions` tables. Run a one-time migration script mapping existing `read_files` permissions to the new format.
- **Migration Risk**: Medium. RLS policies on IPL Finder will need to be updated to recognize both old and new permission formats during the transition.
- **Rollback**: Rely on the old `user_permissions` table.

### Phase 5: Convert IPL Finder
- **Objective**: Make IPL Finder a consumer of `community-platform`.
- **Scope**: Strip out `Admin.tsx`, `LoginScreen.tsx`, and `PendingApprovalScreen.tsx` from IPL Finder. Implement the redirect logic (unauthorized users go to `community.veryresto.com/login`).
- **Migration Risk**: High. This cuts over the primary entry path.
- **Rollback**: Revert the commit in IPL Finder to restore its native login screens.

### Phase 6: Integrate Rekap
- **Objective**: Prove the multi-app model by onboarding `rekap.veryresto.com`.
- **Scope**: Register Rekap in the `applications` table, define `rekap.view` permissions, and use the shared session.
- **Migration Risk**: Low (for existing apps).

## 3. Shared Database Strategy

The single Supabase project will contain both shared identity tables and app-specific tables.

### Shared Platform Tables (Identity Ownership)
- `profiles` (User metadata, including `approval_status` enum: `pending`, `approved`, `suspended`, `rejected`)
- `user_roles` (Global roles like Super Admin)
- `applications` (Registry of sibling apps)
- `app_permissions` (Valid capabilities)
- `user_app_permissions` (Grants)

### App-Specific Tables (App Ownership)
- `files` (Owned by IPL Finder)
- `transactions` (Owned by Kas)

### Permission Namespacing Strategy
To prevent naming collisions and clarify intent, permissions must be explicitly namespaced by the consuming application:
- `ipl_finder.read`
- `ipl_finder.upload`
- `rekap.view`
- `kas.manage`
- `surat.create`

## 4. Shared Permission Architecture

To support multiple applications, the generic `user_permissions` table must be replaced with a robust capability assignment model.

### New Table Schema
```sql
-- Registry of ecosystem applications
CREATE TABLE applications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug TEXT UNIQUE NOT NULL, -- e.g., 'ipl_finder'
  name TEXT NOT NULL,
  url TEXT NOT NULL
);

-- Valid capabilities defined by each app
CREATE TABLE app_permissions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  app_id UUID REFERENCES applications(id),
  permission_key TEXT NOT NULL, -- e.g., 'upload'
  description TEXT,
  UNIQUE(app_id, permission_key)
);

-- Assignment of capabilities to users
CREATE TABLE user_app_permissions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  app_permission_id UUID REFERENCES app_permissions(id),
  granted_by UUID REFERENCES auth.users(id),
  granted_at TIMESTAMP WITH TIME ZONE DEFAULT now(),
  UNIQUE(user_id, app_permission_id)
);
```

### RLS Evolution
Row Level Security in consumer apps (e.g., IPL Finder) will evolve to query these new tables. The `has_permission` security definer function will be updated to accept the namespaced format:
```sql
-- In IPL Finder's RLS
USING ( has_namespaced_permission(auth.uid(), 'ipl_finder.upload') )
```

## 5. Authentication & Session Strategy

Since all apps operate under the `.veryresto.com` root domain, they intended to share the Supabase Auth session via cookies.

### Subdomain Session Sharing
- Supabase Auth can be configured to write cookies scoped to the root domain (`Domain=.veryresto.com`).
- This allows a token minted on `community.veryresto.com` to be automatically sent to the Supabase API when the user navigates to `rekap.veryresto.com`.

### Redirect / Login Flow
1. User visits `ipl-finder.veryresto.com`.
2. The app detects no valid session (or local missing permissions).
3. Redirects to `community.veryresto.com/login?redirect=https://ipl-finder.veryresto.com`.
4. User authenticates on `community-platform`.
5. `community-platform` checks global profile completion (Waiting Room) and ensures the user is globally approved.
6. `community-platform` redirects the browser back to `https://ipl-finder.veryresto.com`.
7. `ipl-finder.veryresto.com` sees the shared session cookie, initializes its local state, and queries the user's `ipl_finder.*` permissions.

### Fallback Redirect Session Strategy
In scenarios where shared cookies become inconsistent across subdomains (e.g., due to strict browser tracking protections like ITP), the architecture must fall back gracefully to ensure continuous access:
- **Central Login Page Interstitial**: If an app fails to read the session cookie, it redirects to the central `community.veryresto.com` login page.
- **Token Refresh Redirect**: If the central auth server detects an active session despite the consumer app's missing cookie, it immediately redirects back to the consumer app with a short-lived token or authorization code in the URL.
- **Lightweight SSO Handoff**: The consumer app exchanges this URL parameter for a local session, effectively establishing a lightweight SSO flow without relying solely on cross-domain cookies.

## 6. Governance Model

### Resident Lifecycle

To explicitly separate global community trust from app-specific capability assignments, a user's global standing will be tracked via a new `approval_status` enum column on the `profiles` table.

1. **Unregistered**: Unknown to the system.
2. **Pending** (`profiles.approval_status = 'pending'`): Authenticated via Google, `profiles` record created, Waiting Room data collected, but lacks global approval.
3. **Approved (Global)** (`profiles.approval_status = 'approved'`): Admin grants baseline platform trust, establishing the user as a verified community member.
4. **Provisioned (App-Specific)**: Admin grants specific app permissions (e.g., `ipl_finder.read`) to an `approved` user.
5. **Suspended/Rejected** (`profiles.approval_status = 'suspended'` or `'rejected'`): Global flag preventing access to any app, instantly overriding all local app permissions.

### Admin Delegation
- **Global Admins** (`user_roles.role = 'admin'`): Can approve global residents, modify system parameters, and assign permissions across all apps.
- **App Admins** (Future consideration): Users with `[app_slug].admin` permission can assign access *only* for their specific application's capabilities.

## 7. Migration Safety

To guarantee IPL Finder remains fully operational, a **strangler fig pattern** will be employed.

### Compatibility Strategy
- During Phases 1-4, `community-platform` will be deployed alongside IPL Finder.
- `community-platform` will use the *exact same* Supabase backend.
- We will temporarily support **both** the old `user_permissions` table and the new `user_app_permissions` table. 
- A database trigger will keep the two tables synchronized during the transition window to ensure IPL Finder's existing RLS policies don't break.

### Fallback Approach
- If the session sharing mechanism faces SameSite/cookie issues in certain browsers during Phase 5 testing, we can revert IPL Finder to use its own native login screen temporarily without corrupting the underlying database state.

## 8. App Onboarding Model

Future developers adding new apps (e.g., `surat.veryresto.com`) must follow this standard integration checklist:

1. **Register the App**: Insert a record into the `applications` table via the `community-platform` dashboard.
2. **Define Permissions**: Register required capabilities (e.g., `surat.write`) in `app_permissions`.
3. **Configure Environment**: 
   - Point the app to the shared Supabase URL and Anon Key.
   - Configure DNS for `[app-name].veryresto.com`.
4. **Implement Auth Hook**: Use standard middleware/hooks to redirect unauthenticated traffic:
   ```javascript
   if (!session) window.location.href = `https://community.veryresto.com/login?redirect=${window.location.href}`
   ```
5. **Write App-Specific RLS**: Secure the app's isolated tables using the shared `has_namespaced_permission` function.

## 9. Deployment Strategy

- **Topology**: All applications (including `community-platform`) will be deployed as independent services on Fly.io.
- **Database**: A single, shared Supabase instance manages the Postgres database, Auth, and Storage.
- **Environment Variables**: `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` will be identical across all applications in the ecosystem.
- **DNS**: Cloudflare/DNS provider must route subdomains correctly (`community.`, `rekap.`, `ipl-finder.`) to their respective Fly.io applications.

---

## 10. Constraints

- **No Code Implementation Yet**: This document is strictly an implementation strategy. No Next.js/Vite scaffolding has been created.
- **No Refactoring Yet**: The existing IPL Finder codebase remains untouched.
- **No New Migrations Yet**: The SQL snippets above are design proposals, not executed migrations.
- **No Enterprise IAM**: We are relying on simple Postgres tables and Supabase Auth, avoiding complex distributed OAuth/OIDC providers (like Auth0 or Keycloak) to maintain operational simplicity for the community.
