# Community Platform Implementation Plan

This document defines the strategy for safely and incrementally extracting the identity, approval, and permission system from the IPL Finder codebase into a dedicated, reusable shared community identity platform (`community-platform`).

## 1. Target Architecture

The future architecture relies on a centralized identity provider sharing a Supabase backend with multiple independent applications.

### Application Ownership Model
To ensure future residents can safely build their own apps and teams can evolve independently, there is a strict separation of concerns:

**Applications are responsible for:**
- defining their own capabilities (under their namespace)
- maintaining their own business logic
- securing their own tables via RLS

**The platform (`community-platform`) is responsible for:**
- assigning capability templates/roles to users
- resident approval and lifecycle
- global governance and moderation

### Shared Platform: `community-platform` (`community.veryresto.com`)
This application acts as the single source of truth for identity and governance.
**Responsibilities:**
- **Authentication Entrypoint**: Centralized login via Google OAuth.
- **Waiting Room**: Collects mandatory profile information (e.g., house number) for new users.
- **Resident Approval**: The global dashboard where admins approve/reject new residents.
- **Permission Assignment**: The governance center to assign app-specific roles (e.g., granting a user `ipl_finder:admin`).
- **Shared Profiles**: The master source for user avatars, names, contact methods, and global standing (`approval_status`).

### Independent Applications (e.g., IPL Finder, Rekap, Surat, Kas)
Each consumer application remains fundamentally autonomous.
**Responsibilities:**
- **Independently Deployable**: Own separate codebases, repositories, and CI/CD pipelines.
- **Shared Session**: Seamlessly consume the Supabase session generated by `community-platform` via subdomain cookie sharing (`*.veryresto.com`).
- **Centralized Identity**: Read from the shared `profiles` and `user_app_roles` tables for UI state and RLS enforcement.

## 2. Extraction Strategy

To prioritize minimal disruption and operational safety, the extraction will follow a phased approach, temporarily tolerating data duplication to avoid a "big bang" migration.

### Phase 1: Scaffold `community-platform`
- **Objective**: Create the new repository and deploy the baseline framework.
- **Scope**: Next.js or Vite React scaffold, Tailwind CSS, Supabase client connection pointing to the *existing* IPL Finder Supabase project.
- **Migration Risk**: None. Sits safely alongside the production IPL Finder.

### Phase 2: Move Waiting-Room UI
- **Objective**: Implement the `PendingApprovalScreen` and `RejectedScreen` in `community-platform`.
- **Scope**: Recreate the UI for gathering `house_number` and displaying rejection status.
- **Migration Risk**: Low. These screens run parallel to the ones in IPL Finder.

### Phase 3: Move & Expand Approval/Admin UI
- **Objective**: Create a scalable Global Admin Dashboard in `community-platform` that explicitly separates global platform powers from local app powers.
- **Scope**: 
  - **Global Governance Tab**: UI to manage pending residents (Approve/Reject/Suspend) and assign platform-wide roles (`global_admin`, `resident_verifier`). Must enforce requiring a *Reason* (saved to the audit trail) for negative actions.
  - **App Governance Tab**: UI to assign users to *App Role Templates* (e.g., "Assign to IPL Finder Admin"). Directly interacting with raw permissions (e.g., `upload`, `read`) will be abstracted away from the assignment UI to prevent interface explosion.
  - **Audit Logging**: Implement writes to `governance_events` for every action.
- **Migration Risk**: Low. New roles run in parallel.

### Phase 4: Centralize Permissions (App-RBAC)
- **Objective**: Introduce the scalable Role-Based Access Control (RBAC) architecture.
- **Scope**: Create `applications`, `app_permissions`, `app_roles`, `app_role_permissions`, and `user_app_roles` tables. Introduce the legacy synchronization trigger.
- **Migration Risk**: Medium. Legacy triggers will ensure IPL Finder's existing RLS policies don't break during the transition.

### Phase 5: Convert IPL Finder
- **Objective**: Make IPL Finder a consumer of `community-platform`.
- **Scope**: Strip out `Admin.tsx`, `LoginScreen.tsx`, and `PendingApprovalScreen.tsx` from IPL Finder. Update all RLS policies to use the new `has_namespaced_permission()` helper. Implement the redirect logic for unauthenticated users.
- **Migration Risk**: High. This cuts over the primary entry path.

### Phase 6: Integrate Rekap
- **Objective**: Prove the multi-app model by onboarding `rekap.veryresto.com`.
- **Scope**: Register Rekap in the `applications` table, define `rekap.view` permissions mapped to an app role, and use the shared session.

## 3. Shared Database Strategy

The single Supabase project will contain both shared identity tables and app-specific tables.

### Shared Platform Tables (Identity Ownership)
- `profiles` (User metadata, including `approval_status` enum: `pending`, `approved`, `suspended`, `rejected`)
- `user_roles` (Global platform roles like `global_admin`, `platform_moderator`)
- `governance_events` (Immutable audit trail for all governance actions)
- `applications` (Registry of sibling apps)
- `app_permissions` (Raw capabilities defined by apps)
- `app_roles` (App-specific role templates mapping to capabilities)
- `app_role_permissions` (Join table for templates and capabilities)
- `user_app_roles` (Assignment of templates to users)

### Permission Namespacing Strategy
To prevent naming collisions and clarify intent, permissions must be explicitly namespaced by the consuming application:
- `ipl_finder.read`
- `ipl_finder.upload`
- `rekap.view`

## 4. Shared Permission Architecture (App-RBAC)

To prevent governance UI complexity from exploding, we will use a Role-Based Access Control (RBAC) model at the app level. Admins assign users to *Roles/Templates*, and the database dynamically expands these into raw capabilities for RLS.

### New Table Schema
```sql
-- Audit Trail
CREATE TABLE governance_events (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  actor_user_id UUID REFERENCES auth.users(id),
  target_user_id UUID REFERENCES auth.users(id),
  action TEXT NOT NULL, -- e.g., 'approved_resident', 'suspended_resident', 'granted_app_role'
  reason TEXT, -- Operational context (e.g., 'Suspended due to move out')
  metadata JSONB,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT now()
);

-- Registry of ecosystem applications
CREATE TABLE applications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug TEXT UNIQUE NOT NULL, -- e.g., 'ipl_finder'
  name TEXT NOT NULL,
  url TEXT NOT NULL
);

-- Raw capabilities defined by each app
CREATE TABLE app_permissions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  app_id UUID REFERENCES applications(id),
  permission_key TEXT NOT NULL, -- e.g., 'upload'
  description TEXT,
  UNIQUE(app_id, permission_key)
);

-- App Roles / Templates (e.g. App: 'ipl_finder', Role: 'admin')
CREATE TABLE app_roles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  app_id UUID REFERENCES applications(id),
  name TEXT NOT NULL, -- e.g., 'admin', 'resident', 'verifier'
  description TEXT,
  UNIQUE(app_id, name)
);

-- Maps individual capabilities to App Roles/Templates
CREATE TABLE app_role_permissions (
  app_role_id UUID REFERENCES app_roles(id),
  app_permission_id UUID REFERENCES app_permissions(id),
  PRIMARY KEY (app_role_id, app_permission_id)
);

-- Assignment of Roles/Templates to Users
CREATE TABLE user_app_roles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  app_role_id UUID REFERENCES app_roles(id),
  granted_by UUID REFERENCES auth.users(id),
  granted_at TIMESTAMP WITH TIME ZONE DEFAULT now(),
  UNIQUE(user_id, app_role_id)
);
```

### RLS Evolution & Capability Expansion
Consumer apps will query raw capabilities in their RLS to remain highly modular. The backend dynamically resolves templates into granular capabilities:
```sql
-- Checks if the user is assigned a template/role that contains the requested capability
CREATE OR REPLACE FUNCTION has_namespaced_permission(uid UUID, namespaced_perm TEXT)
RETURNS BOOLEAN AS $$
BEGIN
  RETURN EXISTS (
    SELECT 1 
    FROM user_app_roles uar
    JOIN app_roles ar ON uar.app_role_id = ar.id
    JOIN applications app ON ar.app_id = app.id
    JOIN app_role_permissions arp ON arp.app_role_id = ar.id
    JOIN app_permissions ap ON arp.app_permission_id = ap.id
    WHERE uar.user_id = uid
      AND (app.slug || '.' || ap.permission_key) = namespaced_perm
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

## 5. Authentication & Session Strategy

Since all apps operate under the `.veryresto.com` root domain, they intended to share the Supabase Auth session via cookies.

### Subdomain Session Sharing
- Supabase Auth can be configured to write cookies scoped to the root domain (`Domain=.veryresto.com`).
- This allows a token minted on `community.veryresto.com` to be automatically sent to the Supabase API when the user navigates to `rekap.veryresto.com`.

### Fallback Redirect Session Strategy
In scenarios where shared cookies become inconsistent across subdomains (e.g., due to strict browser tracking protections like ITP), the architecture must fall back gracefully to ensure continuous access:
- **Central Login Page Interstitial**: If an app fails to read the session cookie, it redirects to the central `community.veryresto.com` login page.
- **Token Refresh Redirect**: If the central auth server detects an active session despite the consumer app's missing cookie, it immediately redirects back to the consumer app with a short-lived token or authorization code in the URL.
- **Lightweight SSO Handoff**: The consumer app exchanges this URL parameter for a local session, effectively establishing a lightweight SSO flow.

## 6. Governance Model

### Soft Delete Principle
Destructive removal (deletion) of residents is strictly prohibited. Governance relies entirely on `profiles.approval_status` (suspended/rejected) and zeroed app permissions, ensuring historical records and event references are never orphaned.

### App Capability Ownership
To support a decentralized app ecosystem later:
- **App Registration**: Global Admins register new applications and assign an immutable `slug` (e.g., `rekap`).
- **Capability Definition**: App developers have autonomy to self-register new capabilities (e.g., `rekap.export`) and templates *only* within their assigned `slug` namespace.

### Resident Lifecycle
To explicitly separate global community trust from app-specific capability assignments, a user's global standing will be tracked via the `approval_status` enum column on the `profiles` table.

1. **Unregistered**: Unknown to the system.
2. **Pending** (`profiles.approval_status = 'pending'`): Authenticated, Waiting Room data collected, lacks global approval.
3. **Approved (Global)** (`profiles.approval_status = 'approved'`): Admin grants baseline platform trust, establishing the user as a verified community member.
4. **Provisioned (App-Specific)**: Admin grants specific app role templates (e.g., `ipl_finder:admin`) to an `approved` user.
5. **Suspended/Rejected** (`profiles.approval_status = 'suspended'` or `'rejected'`): Global flag preventing access to any app, instantly overriding all local app permissions.

### Admin UX Distinction
The Admin UI must never blur resident verification with app provisioning. 
**Example UX Requirement:**
```text
Resident Status: 🟢 Approved Resident
-----------------------------------
App Access: 
✓ Rekap (Role: Viewer)
✓ IPL Finder (Role: Admin)
✗ Kas (No Access)
```

## 7. Migration Safety & Sunset Strategy

To guarantee IPL Finder remains fully operational, a **strangler fig pattern** will be employed using a Legacy Synchronization Trigger.

### Compatibility Strategy
- During Phases 1-4, `community-platform` will be deployed alongside IPL Finder.
- We will temporarily support **both** the old `user_permissions` table and the new `user_app_roles` structures. 
- A database trigger will keep the two environments synchronized during the transition window to ensure IPL Finder's existing RLS policies don't break.

### Sunset Strategy (Removal Milestone)
This compatibility glue is powerful but dangerous if semantics diverge later. It must not become permanent architecture.
- **Sunset Milestone**: 7 days post-deployment of **Phase 5 (Convert IPL Finder)**. Once IPL Finder is fully updated to query the new `has_namespaced_permission` helper in production, the legacy tables are officially obsolete.
- **Removal Actions**: The legacy sync triggers (`sync_legacy_permissions_trg`) and legacy tables (`user_permissions`) will be explicitly dropped, completing the migration.

### Permission Source Architecture
To prevent architectural confusion, we establish a strict separation of roles:
- **Canonical Permission Source**:
  * `applications`
  * `app_roles`
  * `app_permissions`
  * `user_app_roles`
  All administration, role assignments, and platform capability mappings are performed exclusively on these canonical tables. They are the single source of truth.
- **Compatibility Projection**:
  * `user_permissions` (ONLY)
  This table is treated strictly as a **read-only compatibility projection** of the canonical source. It is populated solely by database triggers to support legacy applications (like IPL Finder) during the migration window. No direct writes should ever target this table.

## 8. App Onboarding Model

Future developers adding new apps must follow this standard integration checklist:

1. **Register the App**: Insert a record into the `applications` table.
2. **Define Permissions & Roles**: Register capabilities in `app_permissions` and group them into templates in `app_roles`.
3. **Configure Environment**: Point the app to the shared Supabase URL and configure DNS for `[app-name].veryresto.com`.
4. **Implement Auth Hook**: Use standard middleware to redirect unauthenticated traffic to `community.veryresto.com`.
5. **Write App-Specific RLS**: Secure the app's isolated tables using the shared `has_namespaced_permission` function.

---

## 9. Deployment Strategy

- **Topology**: All applications will be deployed as independent services on Fly.io.
- **Database**: A single, shared Supabase instance manages the Postgres database, Auth, and Storage.
- **Environment Variables**: `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` will be identical across all applications.

---

## 10. Constraints

- **No Enterprise IAM**: We are relying on simple Postgres tables and Supabase Auth, avoiding complex distributed OAuth/OIDC providers (like Auth0 or Keycloak) to maintain operational simplicity for the community.
